# Submission Response Matrix

## Homepage Form
### If the submission is vague
Use:

> Thanks for reaching out. The cleanest next step is to isolate the bottleneck first so I do not guess at the build. If you want, I can route this into a short async teardown and point to the first change I would make.

### If the bottleneck is already clear
Use:

> This already sounds specific enough to scope. I can map the smallest pilot build I would start with and keep the process async.

## Async Teardown
### If the submission is a strong fit
Use:

> I reviewed the issue and the first thing I would tighten is the front-end path between interest and follow-up. I can send the teardown first, then scope the pilot if it matches what you need.

### If the submission is too broad
Use:

> There are a few different issues mixed together here. The fastest way to keep this useful is to choose the single bottleneck you want fixed first, then I can scope the pilot around that.

## Pilot Build
### If the submission is ready for scope
Use:

> This is specific enough for a pilot build. I’m going to keep the scope narrow so it ships quickly and fixes the main leak first.

### If the submission still needs narrowing
Use:

> Before I scope the pilot, I want to confirm the exact front-end bottleneck we’re fixing first. Once that is locked, I can send the proposal without turning this into a giant project.

## Reply Deck
### If the submission is product-fit only
Use:

> This is a good fit for the Reply Deck lane. I’ll keep it in the product bucket and use it to shape the paid version rather than forcing it into a service project.

### If the submission sounds like a service lead
Use:

> This actually sounds closer to a Battle Labs service fit than a simple product fit. If you want, I can route it into a teardown or pilot scope instead.
